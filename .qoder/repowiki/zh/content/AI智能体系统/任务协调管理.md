## 架构总览
系统采用“智能体-任务-驱动器-槽位”的分层架构。智能体通过任务管理器创建与等待验证，任务管理器将任务委派给调度器，调度器根据统一配置选择合适的驱动器执行，并通过运行槽位进行并发控制与资源隔离。

```mermaid
sequenceDiagram
participant Agent as "智能体"
participant TM as "任务管理器"
participant Sched as "调度器"
participant Pipe as "流水线处理器"
participant Driver as "视觉/异步驱动器"
participant Slot as "运行槽位"
Agent->>TM : 创建任务/提交验证
TM->>Sched : 分配任务并设置上下文
Sched->>Pipe : 触发流水线步骤
Pipe->>Driver : 选择驱动器执行
Driver->>Slot : 申请/占用槽位
Slot-->>Driver : 槽位可用/拒绝
Driver-->>Pipe : 执行结果/错误
Pipe-->>Sched : 步骤完成/异常
Sched-->>TM : 任务状态更新
TM-->>Agent : 返回结果/等待用户输入
```

## 详细组件分析

### 任务管理器与AskUser混合器
- 作用：封装任务创建、验证请求创建、阻塞等待用户响应、失败计数与保护性停问、返回用户输入与元数据。
- 关键流程：
  - 参数校验：question必填、options非空且为列表
  - 创建验证请求：通过任务管理器创建ask_user验证记录
  - 阻塞等待：在指定超时时间内等待用户确认
  - 失败保护：超过最大连续失败次数后禁止继续提问
  - 结果返回：成功时返回用户输入与验证元数据，失败时返回错误信息

```mermaid
flowchart TD
Start(["进入 ask_user"]) --> CheckArgs["校验 question 与 options"]
CheckArgs --> ArgsOK{"参数有效？"}
ArgsOK --> |否| ReturnErr["返回错误"]
ArgsOK --> |是| CreateVerif["创建验证请求"]
CreateVerif --> WaitResp["等待用户响应最多5分钟"]
WaitResp --> RespOK{"响应成功？"}
RespOK --> |否| IncFail["增加连续失败计数"]
IncFail --> MaxFail{"超过最大失败次数？"}
MaxFail --> |是| DisableAsk["禁用继续提问并返回错误"]
MaxFail --> |否| ReturnFail["返回失败与错误信息"]
RespOK --> |是| ResetFail["重置失败计数"]
ResetFail --> ReturnOK["返回用户输入与验证元数据"]
```

### 统一配置与算力策略
- 任务级算力覆盖优先于实现方配置，用户偏好实现方优先于默认实现。
- 修饰符（modifiers）与数据库配置合并，数据库优先覆盖默认值；最终结果向上取整。
- 前端配置接口扩展：返回任务算力映射与修饰符信息，便于前端展示与选择。

## 性能考虑
- 缓存与截断：会话存储支持内存缓存与消息截断，减少数据库压力与上下文膨胀。

- 统计缓存：定时聚合统计，降低高频查询对数据库的影响。

## 故障排查指南
- AskUser混合器常见问题：
  - 未配置task_manager或task_id：检查智能体初始化与任务绑定。
  - 连续失败超过阈值：系统自动禁用继续提问，需等待用户主动消息。
  - 超时未响应：确认等待超时时间与网络状况。
- 会话清理与过期：
  - 过期会话会被标记为不活跃，检查清理任务与时间阈值。
  - 查询活跃会话时排除不活跃状态。
- 算力计算异常：
  - 任务级覆盖优先：若任务配置存在，实现方配置被忽略。
  - 用户偏好实现方：优先使用用户偏好的实现方算力。
  - 修饰符合并：数据库修饰符优先覆盖默认值。
- 驱动器与槽位：
  - 槽位不足：检查当前占用与实现方并发限制。
  - 错误处理：查看驱动器异常定义与日志输出。
