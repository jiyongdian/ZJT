# 算力多维度计算方案设计

## 1. 背景

当前算力计算仅基于「模型 + 时长(duration)」两个维度，无法覆盖以下场景：
- Kling 视频首尾帧模式需要比首帧模式更高算力
- 高分辨率图片/视频需要更高算力

核心问题：`get_computing_power(duration)` 的签名只接受 duration 参数，无法传入其他影响算力的属性。

## 2. 方案：引入 `power_modifiers`（算力修饰符）机制

### 2.1 核心思路

不改变现有 `{duration: power}` 的基础算力结构，在其上叠加**乘数修饰符**，根据额外属性动态调整算力。

**算力计算公式：** `final_power = ceil(base_power(duration) × (modifier1 × modifier2 × ...))`

**重要：** 先累积所有乘数的乘积，然后对最终结果进行**一次向上取整（ceiling）**，确保结果为整数。这样可以避免浮点数精度问题和多次取整造成的精度损失。

### 2.2 为什么选择修饰符而非多维度矩阵

| 方案 | 优点 | 缺点 |
|------|------|------|
| 多维矩阵 `{duration: {mode: {resolution: power}}}` | 直观 | 组合爆炸，数据库配置极难维护 |
| 修饰符（本方案） | 扩展灵活，配置简单，向后兼容 | 需要定义修饰符规则 |

### 3.1 新增 `PowerModifier` 数据类

```python
@dataclass
class PowerModifier:
    """算力修饰符 - 根据额外属性调整算力"""
    attribute: str              # 属性名，如 'image_mode', 'resolution'
    values: Dict[str, float]    # 属性值 -> 乘数，如 {'first_last_with_tail': 1.5, 'first_last_frame': 1.0}
    default: float = 1.0        # 未匹配时的默认乘数
```

### 3.3 扩展 `get_computing_power` 方法签名

**文件:** `config/unified_config.py` — `UnifiedTaskConfig.get_computing_power()`

新增 `context: Optional[Dict[str, Any]] = None` 参数：

```python
import math

def get_computing_power(
    self,
    duration: Optional[int] = None,
    implementation: Optional[str] = None,
    context: Optional[Dict[str, Any]] = None  # 新增
) -> int:
    # 1. 现有逻辑获取基础算力（不变）
    base_power = ...  # 现有 duration + implementation 逻辑

    # 2. 应用修饰符（先累积乘数，最后一次向上取整）
    if context and self.power_modifiers:
        multiplier = 1.0
        for modifier in self.power_modifiers:
            attr_value = context.get(modifier.attribute)
            if attr_value and attr_value in modifier.values:
                multiplier *= modifier.values[attr_value]
            else:
                multiplier *= modifier.default
        base_power = math.ceil(base_power * multiplier)

    return int(base_power)
```

**关键点：** 
- 先累积所有修饰符的乘数（相乘）
- 最后一次性调用 `math.ceil()` 进行向上取整
- 避免多次 ceiling 导致的精度损失

同样扩展 `ImplementationConfig.get_computing_power()` 签名（预留接口，暂不实现修饰符逻辑）。

**新增 `get_power_modifiers_map()` 方法**（`UnifiedConfigRegistry`）：
```python
@classmethod
def get_power_modifiers_map(cls) -> Dict[int, List[Dict]]:
    """获取 task_id -> power_modifiers 映射"""
    result = {}
    for c in cls._configs.values():
        if c.power_modifiers:
            result[c.id] = [
                {'attribute': m.attribute, 'values': m.values, 'default': m.default}
                for m in c.power_modifiers
            ]
    return result
```

### 3.6 `ai_tools` 表：context 的存储与恢复

**关键发现：** `ai_tools` 表已有字段可以存储 context 信息：
- `extra_config` (JSON) — 已存有 `image_mode`
- `image_size` — 已存有分辨率

**存储（提交任务时）：** 已经在存储了，无需修改。

**恢复（退还算力时）：** 需要从 `task_record` 重建 context。新增辅助函数：

**文件:** `utils/computing_power.py`

```python
def build_context_from_task_record(task_record) -> Dict[str, Any]:
    """从 ai_tools 记录构建算力计算所需的 context"""
    context = {}

    # 1. image_mode：从 extra_config JSON 解析
    if task_record.extra_config:
        try:
            extra = json.loads(task_record.extra_config) if isinstance(task_record.extra_config, str) else task_record.extra_config
            if 'image_mode' in extra:
                # 判断是否有尾帧：first_last_frame 模式下 image_path 有2张图
                image_mode = extra['image_mode']
                if image_mode == 'first_last_frame' and task_record.image_path:
                    paths = task_record.image_path.split(',')
                    if len(paths) > 1:
                        context['image_mode'] = 'first_last_with_tail'
                    else:
                        context['image_mode'] = image_mode
                else:
                    context['image_mode'] = image_mode
        except (json.JSONDecodeError, TypeError):
            pass

    # 2. resolution：从 image_size 字段
    if task_record.image_size:
        context['resolution'] = task_record.image_size

    return context
```

### 3.9 前端 `task_config.js` 适配

**文件:** `web/js/task_config.js`

```javascript
function getComputingPower(taskTypeIdOrModelKey, duration, context = {}) {
    let task;
    // ... 获取 task ...

    const power = task.computing_power;
    let basePower;
    if (typeof power === 'object' && power !== null) {
        basePower = power[duration] || power[Object.keys(power)[0]] || 0;
    } else {
        basePower = power || 0;
    }

    // 应用修饰符（先累积乘数，最后一次向上取整）
    if (task.power_modifiers && context) {
        let totalMultiplier = 1.0;
        for (const modifier of task.power_modifiers) {
            const attrValue = context[modifier.attribute];
            const multiplier = attrValue && modifier.values[attrValue]
                ? modifier.values[attrValue]
                : (modifier.default || 1.0);
            totalMultiplier *= multiplier;
        }
        basePower = Math.ceil(basePower * totalMultiplier);
    }

    return basePower;
}
```

**关键点：** 
- 先累积所有修饰符的乘数（相乘）
- 最后一次性调用 `Math.ceil()` 进行向上取整
- 避免多次 ceiling 导致的精度损失

## 5. 配置示例

### 5.1 Kling 图生视频（id=12）

- 基础算力：`{5: 38, 10: 70}`
- `image_mode` 修饰符：首尾帧 `first_last_with_tail` × 1.5
- 示例（向上取整）：
  - 首帧 5秒 = 38
  - 首尾帧 5秒 = ceil(38 × 1.5) = ceil(57.0) = 57
  - 首帧 10秒 = 70
  - 首尾帧 10秒 = ceil(70 × 1.5) = ceil(105.0) = 105

### 5.2 文生图/图片编辑（分辨率）

- 基础算力：2（1K 基准）
- `resolution` 修饰符：`{'1K': 1.0, '2K': 1.5, '4K': 2.5}`
- 示例（向上取整）：
  - 1K = 2
  - 2K = ceil(2 × 1.5) = ceil(3.0) = 3
  - 4K = ceil(2 × 2.5) = ceil(5.0) = 5

## 6. image_mode 判断逻辑

前端和后端需要统一判断"是否有尾帧"的逻辑：

| 场景 | image_mode 值 | 图片数量 | 算力 context |
|------|--------------|---------|-------------|
| 首帧模式（1张图） | `first_last_frame` | 1 | `first_last_frame` (×1.0) |
| 首尾帧模式（2张图） | `first_last_frame` | 2 | `first_last_with_tail` (×1.5) |
| 多参考图模式 | `multi_reference` | 1-5 | `multi_reference` (×1.0) |
| 首尾帧+参考图 | `first_last_with_ref` | 2+ | `first_last_with_tail` (×1.5) |

存储到 `extra_config` 时，保持原始 `image_mode` 值。判断有无尾帧在计算算力时动态进行：
- 前端：看 `files.length`
- 后端提交时：看 `main_image_list.length`
- 后端退还算力时：看 `image_path.split(',').length`

## 7. 向后兼容

- `context` 参数默认 `None`，不传时行为与现有一致
- 数据库 `power_config` JSON 中无 `modifiers` 字段时，不应用任何修饰符
- 前端 `getComputingPower` 不传 context 时行为不变
- 所有现有调用点无需立即修改，只需在需要额外维度的场景传入 context
- `/api/computing-power-config` 接口新增 `task_power_modifiers` 字段，不影响现有字段

## 8. 验证方案

1. **后端单元测试**：验证 `get_computing_power(context={'image_mode': 'first_last_with_tail'})` 返回正确的修饰后算力
2. **前端测试**：切换 imageMode / 分辨率时，算力显示实时变化
3. **端到端测试**：提交 Kling 首尾帧任务，确认扣减的算力与前端显示一致
4. **退还算力**：任务失败退还算力时，从 `ai_tools` 记录重建 context，退回金额 = 扣除金额
5. **数据库兼容**：旧格式 `power_config` 不含 `modifiers` 时正常工作
6. **接口兼容**：`/api/computing-power-config` 新增字段不影响现有前端
7. **向上取整验证**：
   - 验证浮点数乘积能正确向上取整（如 38 × 1.5 = 57，而非 56）
   - 验证前后端向上取整行为一致
   - 验证极端情况：如 100 × 1.01 = 101 而非 100
   - 确保退还算力金额 = 提交时扣减金额（使用相同向上取整逻辑）
